Experiment detail

Can an AI agent update and improve a public site based on traffic data?

Short answer: partly. The agent can edit pages, run checks, publish static updates, measure aggregate visits, and record decisions. It still needs a human for distribution, spend, identity use, feedback collection, customer-impacting changes, and stronger claims.

ActiveLoop rehearsed

TL;DR

This experiment tests a tiny content/site improvement loop: make a public page clearer, verify it, publish it, see aggregate traffic, and use that weak signal to choose the next safe edit.

So far, the agent has completed the static-site operation part and rehearsed the measured improvement loop. A real post-distribution optimization cycle has not happened yet.

What the agent actually did

  • Built and repeatedly simplified the public site.
  • Changed the structure from broad navigation to Home → Experiments → Privacy.
  • Added an experiment-card page so visitors can quickly see what is being tested.
  • Ran named checks before publishing: public-bundle check, workflow sample-output tests, evidence-source checks, private-path probes, and live-site verification.
  • Published sanitized static files to GitHub Pages.
  • Added aggregate-only GoatCounter measurement.
  • Simulated a visitor path from Experiments to the detail page, noticed the worksheet bridge was unclear, and added a clearer link from the experiment story to the evaluator.

What failed or was confusing

  • The first versions tried to explain too many pages and concepts at once.
  • Visitors found the site abstract until the homepage became experiment-card-first.
  • The experiment detail page itself was too abstract and had to be rewritten into this concrete report.
  • Some steps still depend on external accounts or owner approval, so this is not an unsupervised business.
  • The main operational risks are overfitting to low traffic, publishing unclear copy, leaking private files, or treating setup/test visits as real evidence.

What this suggests

Agents look useful today for narrow, reversible content/site operations: draft edits, run checks, publish static changes, and learn from aggregate signals. They are not ready to independently acquire customers, spend money, use identity, collect user data, or make strong public claims without human review.

If a bad edit ships

The site is static and versioned in Git. A bad public edit can be reverted by rolling the public repo back to an earlier commit and re-running live-site verification. This rollback path is part of why this is a safe first experiment.

Next test

Run one small distribution test, wait for an observation window, then see whether the agent can choose and publish one improvement based on real aggregate traffic rather than setup/testing visits. A useful first success criterion: after 5–10 real visits or 7 days, the agent proposes one page edit from aggregate traffic, publishes it after review, and records why it chose that edit.
