Experiment detail

Can an AI agent safely draft, check, and publish small updates to this site?

Short answer: the guarded update process works; outcome learning has not been shown yet. The agent can edit pages, run policy and site-review checks, open reviewed public pull requests, measure aggregate visits, verify the live site, and run predefined repairs for known failures. It still needs a human for outreach, spend, identity use, feedback collection, customer-impacting changes, and stronger claims.

TL;DR

This experiment tests a tiny website-update process: make a public page clearer, verify it with policy and site-review checks, publish through a protected public PR, see aggregate traffic, and use that weak signal to choose the next safe edit.

So far, the guarded publishing path is implemented and verified. Real outcome learning still depends on future passive traffic after changes.

What the agent actually did

In plain English: the agent changed the site, checked that private files were not exposed, kept claims conservative, opened a pull request instead of publishing directly, and verified the live site after publishing.

  • Built and repeatedly simplified the public site.
  • Changed the structure from broad navigation to Home → Experiments → Privacy.
  • Added an experiment-card page so visitors can quickly see what is being tested.
  • Ran named checks before publishing: public-bundle check, workflow sample-output tests, evidence-source checks, policy guard, site-specific reviewers, private-path probes, and live-site verification.
  • Opened protected public PRs instead of pushing directly to public main.
  • Added aggregate-only GoatCounter measurement.
  • Added machine-readable site state, learning ledgers, and a future experiment publishing contract.
  • Added repeatable repair and live rollback steps for known failures.

What failed or was confusing

  • The first versions tried to explain too many pages and concepts at once.
  • Visitors found the site abstract until the homepage became experiment-card-first.
  • The experiment detail page itself was too abstract and had to be rewritten into this concrete report.
  • Some steps still depend on external accounts or owner approval, so this is not an unsupervised business.
  • The main operational risks are overfitting to low traffic, publishing unclear copy, leaking private files, bypassing branch protection, or treating setup/test visits as real evidence.

What this suggests

Agents look useful today for narrow, reversible content/site operations: draft edits, run checks, publish static changes, and learn from aggregate signals. They are not ready to independently acquire customers, spend money, use identity, collect user data, or make strong public claims without human review.

If a bad edit ships

The site is static and versioned in Git. A bad public edit can be reverted by rolling the public repo back to an earlier commit and re-running live-site verification. This rollback path is part of why this is a safe first experiment.

Next test

Run passive observation windows, then see whether the operator can choose and publish one useful, reversible site improvement from aggregate traffic without overfitting. A useful first success criterion: after enough future visits or 7 days, the site proposes one page edit from aggregate traffic, passes policy and site-review checks, opens a protected public PR, and records why it chose that edit.
